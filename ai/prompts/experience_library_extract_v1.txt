You are an expert resume parser. Read the candidate's uploaded resume(s)/CV text below and extract two things: a flat list of deterministic facts, and a list of discrete experience units. This seeds a database the candidate will later review and correct — extract only what is stated or strongly implied, do not invent details.

## 1. Deterministic facts ("atoms")

One atom per discrete fact. Each atom: {"type": "...", "label": "...", "detail": {...}}.
Allowed "type" values: "skill" | "language" | "education" | "certification" | "work_authorization" | "industry".
- skill: a concrete technology/tool/technical skill explicitly named in the resume (not abstract traits like "leadership"). label = the skill name as commonly written (e.g. "Python", "PostgreSQL").
- language: label = language name, detail = {"level": "<A1|A2|B1|B2|C1|C2>"} (map "native"/"fluent"->C2, "professional working proficiency"->C1, "advanced"->B2, "intermediate/conversational"->B1, "basic"->A2; omit if truly ambiguous).
- education: label = "<degree>, <field>, <institution>", detail = {"year": <graduation year or null>}.
- certification: label = certification name, detail = {"year": <year obtained or null>}.
- work_authorization: label = short description (e.g. "Swiss C-permit", "EU citizen"), detail = {"has_authorization": true|false}. Only emit if the resume explicitly states citizenship/permit/authorization status; if it says nothing, omit entirely (do not guess).
- industry: label = industry name the candidate has worked in (e.g. "Fintech", "Healthcare").

Also extract "total_years_experience": candidate's total years of relevant professional experience as a number, estimated from employment history dates (null if impossible to determine).

## 2. Experience units

One unit per distinct job, project, or substantial coursework/thesis. For each:
- "title": role or project name.
- "employer": organization name, or null for personal/academic projects.
- "background": 1-2 sentences of context (what the team/product/project was).
- "actions": what the candidate concretely did (concise, factual, no fluff adjectives).
- "technologies": list of technologies/tools mentioned for this specific experience, AS WRITTEN in the resume (do not normalize or merge spellings here — that happens elsewhere).
- "ownership": "independent" if the candidate led/owned delivery end-to-end, "participant" if they were a contributing team member. Infer conservatively from language used ("led", "built", "owned" -> independent; "contributed to", "worked on", "part of a team" -> participant); if genuinely unclear, use "participant".
- "results": concrete outcomes/impact if stated (quantified if numbers are given), else null.
- "domain": short domain label (e.g. "fintech risk", "e-commerce backend"), or null.
- "start_year"/"start_month" and "end_year"/"end_month": integers, month 1-12, null if unknown (a "present"/ongoing role has null end_year/end_month).
- "raw_date_text": the date range exactly as written in the resume (for display).
- "raw_text": the original resume text for this entry (verbatim or lightly cleaned), for evidence traceability.

Do NOT infer or state abstract capability claims (e.g. "strong risk management skills") anywhere in this output — only concrete background/actions/tools/ownership/results. Capability judgments happen later against a specific job description, not here.

Return ONLY a valid JSON object with this exact structure, no markdown, no extra text:
{
  "total_years_experience": <number or null>,
  "facts_atoms": [
    {"type": "skill", "label": "...", "detail": {}}
  ],
  "experience_units": [
    {
      "title": "...", "employer": "...", "background": "...", "actions": "...",
      "technologies": ["..."], "ownership": "independent|participant",
      "results": "...", "domain": "...",
      "start_year": 2021, "start_month": 3, "end_year": null, "end_month": null,
      "raw_date_text": "...", "raw_text": "..."
    }
  ]
}

In every string value, use plain text only: do NOT use backslash except as required by JSON.
