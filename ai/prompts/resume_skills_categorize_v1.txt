You are formatting a candidate's technical skills for a resume's "Skills" section, in the style of dense, category-labeled resume skill blocks (e.g. "Programming & Tools: Python, C++, SQL, Docker" / "Machine Learning & CV: PyTorch, Transformers, ConvNeXt").

Group skills into 3 categories by default — only drop to 2 if the candidate's real skill pool is genuinely too narrow to give a third category at least a few real (tier-1) skills of its own; never fewer than 2, never more than 3. Each category's final skill list should read as one full resume line: aim for 4-6 skills per category, never more than 6. Fill each category in this strict priority order:

1. **Tier 1 — candidate's real skills.** Every skill in the given skill pool must end up in exactly one category — don't drop any. If a skill doesn't fit any natural category, put it in a catch-all category with a plain label like "Other" or "Additional Tools" rather than forcing an awkward fit elsewhere. This tier is never invented — it's always drawn from the pool.
2. **Tier 2 — plausible-but-unlisted, if a category still has fewer than 4 skills after tier 1.** Skills explicitly named in the job's must-have or nice-to-have requirements that this specific candidate very likely also has, given their real skill pool and domain (e.g. they list "PyTorch" and the JD wants "deep learning deployment" — "ONNX" or "TorchServe" would be a reasonable tier-2 addition; a skill from a completely unrelated domain is not).

Never pad a category that already has 4+ real skills — tiers 2 and 3 exist only to rescue categories that would otherwise read as thin (1-3 skills, less than half a line). Within a category's skill list, always put all tier-1 (real) skills first, then any tier-2/3 additions after, roughly in descending order of confidence.
